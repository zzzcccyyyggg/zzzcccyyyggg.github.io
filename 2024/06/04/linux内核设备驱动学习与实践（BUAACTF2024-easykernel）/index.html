linux内核设备驱动学习与实践（BUAACTF2024-easykernel）

第一章 设备驱动简介

设备驱动的角色就是为用户构建一套标准化的调用映射到作用于实际硬件的和设备相关的操作上

注意驱动程序的机制和策略的分离问题，程序员在编写访问硬件的内核代码时，不应该给用户强加任何特定的策略，驱动程序解决的应该仅仅是如何使硬件可用的问题。（但也不是非常绝对的）

内核的划分

进程管理、内存管理、文件系统、设备控制、网络功能

设备和模块的分类

  • 字符设备(Character Device)
    • 以字节为单位进行数据传输,如串行端口、鼠标键盘等
    • 基于流的设备
  • 块设备(Block Device)
    • 以块为单位进行数据传输,如硬盘、U盘等存储设备
    • 基于缓冲区的设备
  • 网络设备(Network Device)
    • 用于网络通信,如以太网卡等

字符设备与块设备的区别

数据传输方式

  • 字符设备是以字节流的方式进行数据传输,一次传输的数据量没有限制。
  • 块设备则是以固定大小的数据块为单位进行传输,通常一个块为512字节或4KB。

缓冲区使用

  • 字符设备通常不使用缓冲区,数据直接在应用程序和设备驱动程序之间传输。
  • 块设备使用了缓冲区机制,在数据传输时先将数据存储到内核缓冲区,然后再传输到应用程序或写入磁盘。

访问方式

  • 字符设备通常采用线性访问方式,应用程序可以任意定位到设备的任意位置访问数据。
  • 块设备则采用随机访问方式,应用程序需要指定需要访问的块号。

设备实例

  • 字符设备的典型实例有终端设备、串行端口、鼠标键盘等。
  • 块设备的典型实例有硬盘驱动器、U盘、光驱等存储设备。

数据处理

  • 字符设备一般用于数据的传输和控制,不承担数据的存储功能。
  • 块设备主要用于存储大量结构化数据,如文件系统等。

模块参数的作用

模块参数有多种用途,例如:

  1. 调试和测试: 通过调整模块参数,可以方便地测试不同的配置和场景,而无需重新编译模块。
  2. 动态配置: 模块参数可以用于动态配置模块的行为,例如设置缓冲区大小、启用/禁用某些功能等。
  3. 特性开关: 通过布尔型模块参数,可以控制是否启用某些特性或功能。
  4. 自动检测: 模块可以通过检测某些参数的值来自动进行适当的配置和操作。
  5. 生产和调试版本切换: 通过设置不同的模块参数,可以在生产版本和调试版本之间切换。

最佳实践

在使用模块参数时,有一些最佳实践需要注意:

  1. 提供默认值: 为每个模块参数提供合理的默认值,以防止在不传递参数时出现未定义的行为。
  2. 验证参数值: 在使用模块参数之前,应该验证其值是否符合预期范围,以避免出现错误或不当行为。
  3. 文档化: 在模块代码中或其他文档中记录每个模块参数的用途、类型和可能的值范围。
  4. 恰当使用权限: 根据需要设置适当的参数权限,以防止在不应该的时候修改参数值。

通过合理使用模块参数,可以显著提高内核模块的灵活性和可配置性,同时也为内核开发人员提供了更好的调试和测试手段。

首先，字符设备到底是什么呢，字符设备（Character Device）是指在操作系统中以字符为单位进行数据输入和输出的设备。它可以是硬件设备，也可以是虚拟设备。字符设备通过字符流的方式进行数据读写，即按照字符的顺序逐个处理数据，以下是AI给出的字符设备的常见类型：

  1. 终端设备:如控制台(Console)、串行端口(Serial Port)等。
  2. 伪终端设备:用于模拟终端的虚拟设备,如pty设备。
  3. 内存设备:如/dev/null、/dev/zero、/dev/random等。
  4. 网络设备:如网卡、UNIX域套接字等。
  5. 其他设备:如键盘、鼠标、声卡等一些外围设备。

主次编号

在 Linux 中，设备文件通常位于 /dev 目录下。字符设备文件的特殊属性由 ls -l 输出的第一列中的 "c" 标识，而块设备由 "b" 标识。设备文件由主设备号和次设备号标识，主设备号用于标识设备对应的驱动程序，而次设备号由内核使用，用于正确确定设备文件所指的设备。

主设备号通常由内核分配，而次设备号则由驱动程序使用者来管理，以便于区分具体的设备实例。

而设备编号采用，dev_t 类型表达，并且可以通过以下相关宏

设备编号的获取

设备编号可以通过两种方式获取：静态分配和动态分配，静态分配需要提前知道要使用的主设备号，而动态分配则由内核自动分配一个可用的主设备号。

静态分配

  • first：分配的起始设备编号。通常，次设备号设置为0，但也可以是其他值。
  • count：请求的连续设备编号的数量。
  • name：与这些设备编号关联的设备名称，这个名称会显示在 /proc/devices 和 sysfs 中。

如果注册成功，这个函数返回0；如果出错，则返回一个负数错误码。

动态分配

内核动态分配主设备号

  • dev：一个输出参数，函数成功后会保存分配的第一个设备编号。
  • firstminor：请求的第一个次设备号，通常是0。
  • count和name：与register_chrdev_region中的相同。

关键数据结构

这里首先大致了解下Linux是如何管理文件系统，并对其进行相关操作的，Linux通过 inode 和 dentry 结构体来组织文件和目录

  • 索引节点，即 inode，用来记录文件的元数据，比如 inode 编号、文件大小、访问权限、修改日期、数据的位置等。索引节点和文件一一对应，它跟文件内容一样，都会被持久化存储到磁盘中。所以记住，索引节点同样占用磁盘空间。
  • 目录项，即 dentry，用来记录文件的名字、索引节点指针以及与其他目录项的关联关系。多个关联的目录项，就构成了文件系统的目录结构。不过，不同于索引节点，目录项是由内核维护的一个内存数据结构，所以通常也被叫做目录项缓存。

用户在打开文件时，往往经过以下步骤

1. 路径解析

内核通过路径解析（path resolution）在目录树中查找目标文件的 dentry 结构体。路径解析过程涉及多个系统调用，如 open、lookup 等。

2. 获取 inode

一旦找到目标文件的 dentry 结构体，内核通过 dentry 的 d_inode 成员获取对应的 inode 结构体。

3. 创建 file 结构体

内核创建一个新的 file 结构体实例来表示打开的文件。file 结构体包含文件的状态信息和操作函数指针。

这里继续了解下 file 结构体 file 结构体是内核中表示一个打开文件实例的数据结构。每次打开文件时，内核都会创建一个 file 结构体实例来表示该文件。file 结构体包含了文件的状态信息和操作函数指针，那么内核时如何管理file结构体呢，这就涉及到文件描述符（fd）了，每个进程都有一个文件描述符表（file descriptor table），这是一个指针数组，其中每个元素指向一个 file 结构体，当一个文件被打开时，内核会创建一个 file 结构体实例，并在文件描述符表中找到一个空闲的文件描述符，将其指向这个 file 结构体，用户空间程序则通过文件描述符来引用这个 file 结构体，而当用户要对此文件进行相关的操作时，系统调用会根据其fd找到对应的file结构体，然后调用结构题中的操作函数。

file_operations

file_operations 是一个由内核定义的结构体，它包含了指向函数的指针，这些函数由驱动程序实现。这些函数实现了对设备文件的操作，如打开 (open)、读取 (read)、写入 (write)、释放 (release) 等。当用户空间程序对设备文件执行操作时，内核会调用这个结构体中对应的函数。

scull的内存策略

在 scull 字符设备驱动程序中,引入了"量子(quantum)"和"量子集(quantum set)"的概念,用于管理和分配内存

量子(quantum)

量子是 scull 驱动程序中分配内存的基本单位。它是一个固定大小的内存块,可以被设置为不同的值,例如 4096 字节(4KB)或其他值。量子的大小由 scull_dev 结构体中的 quantum 成员变量指定。

量子集(quantum set)

量子集是由一组连续的量子组成的更大的内存块。它由 scull_qset 结构体表示,包含一个指向第一个量子的指针和一个计数器,表示该量子集中包含的量子数量。

在 scull_dev 结构体中,data 成员指向第一个量子集,而 qset 成员表示当前有多少个量子集。

量子与量子集的关系

scull 驱动程序使用量子集来管理内存分配,每个量子集包含多个连续的量子。当需要分配内存时,驱动程序首先尝试在现有的量子集中找到足够大小的空间。如果没有足够的空间,它将分配一个新的量子集。

每个量子集的大小由 qset 变量决定,它表示该集合中包含多少个量子。例如,如果 qset 为 4,并且 quantum 为 4096,那么每个量子集将包含 4 个量子,总共 16KB 的内存空间。

这个脚本是一个用于初始化Linux环境的shell脚本，通常在系统启动时运行，特别是在嵌入式系统或简化的Linux环境中。下面是对该脚本各部分的详细解释：

  1. #!/bin/sh：
    • 这是一个shebang行，告诉系统使用/bin/sh（shell）来解释执行该脚本。
  2. echo "INIT SCRIPT"：
    • 输出“INIT SCRIPT”，作为脚本开始执行的标志。
  3. mkdir /tmp：
    • 创建一个名为/tmp的目录，用于存放临时文件。虽然大多数Linux系统会自动创建此目录，但在某些最小化或自定义的环境中可能需要手动创建。
  4. mount -t proc none /proc：
    • 挂载proc文件系统到/proc目录。proc文件系统是一个伪文件系统，提供了一个接口来访问内核数据结构，通常用于访问系统信息和运行中进程的信息。
  5. mount -t sysfs none /sys：
    • 挂载sysfs文件系统到/sys目录。sysfs文件系统提供了一个访问和管理内核对象信息的接口，比如设备和驱动程序。
  6. mount -t devtmpfs none /dev：
    • 挂载devtmpfs到/dev目录。devtmpfs是一种临时文件系统，用于管理设备节点，让用户空间程序可以访问硬件设备。
  7. mount -t debugfs none /sys/kernel/debug：
    • 挂载debugfs文件系统到/sys/kernel/debug。debugfs是一个面向调试的文件系统，提供了一个窗口来访问Linux内核的调试信息。
  8. mount -t tmpfs none /tmp：
    • 再次尝试以tmpfs类型挂载/tmp目录。tmpfs是一种基于内存的文件系统，可以提供快速的文件访问速度。这行看起来与第三行重复，实际上可能是为了确保/tmp是以tmpfs方式挂载的，这样做可以提高速度并减少磁盘I/O。
  9. echo -e "Boot took $(cut -d' ' -f1 /proc/uptime) seconds"：
    • 输出系统启动耗时。/proc/uptime文件包含了系统启动以来的时间（秒）。cut -d' ' -f1命令用于提取该时间。
  10. setsid /bin/cttyhack setuidgid 1000 /bin/sh：
    • setsid创建一个新的会话，并使得/bin/cttyhack setuidgid 1000 /bin/sh在这个新会话中运行。cttyhack是busybox中的一个工具，用于在没有控制终端的情况下启动shell。setuidgid设置用户和组ID为1000，然后执行/bin/sh，这通常用于以非root用户的身份启动shell。

console=ttyS0

console=ttyS0 是一个内核启动参数，用于指定系统的控制台输出设备。在这个上下文中，ttyS0 通常指的是第一个串行端口，这是历史上计算机通用的串行通信接口。

  • 作用：这个参数的作用是将系统消息和控制台输出重定向到ttyS0串行端口。这对于无头服务器（没有显示器的服务器）或通过远程终端访问系统非常有用，因为它允许开发者或管理员通过串行连接查看启动日志、内核消息以及与系统交互。
  • 应用场景：在虚拟化环境中，如QEMU，console=ttyS0可以让虚拟机的控制台输出通过宿主机的终端或特定的控制台窗口进行访问，这对于调试和管理虚拟机尤其重要。
  • 优点：使用串行控制台可以简化系统日志的收集和远程管理，特别是在图形界面不可用或不希望使用图形界面的情况下。

root=/dev/ram

root=/dev/ram 是另一个内核启动参数，用于指定系统的根文件系统位置。

  • 作用：这个参数告诉内核使用内存中的一个RAM磁盘作为根文件系统。/dev/ram 表示一个虚拟的RAM设备，它利用系统内存来模拟硬盘驱动器。
  • 应用场景：这种配置常见于需要快速启动和运行的环境，如嵌入式系统、恢复模式、或者某些特定的测试环境。使用RAM磁盘可以显著提高文件访问速度，因为它避免了物理磁盘的I/O延迟。
  • 优点：由于RAM磁盘在内存中，这使得对文件系统的读写操作非常快。这对于性能要求高的应用或者在只读存储介质上启动系统时非常有用（例如，从只读的光盘启动Linux发行版）。
  • 注意点：虽然使用RAM磁盘可以提高速度，但由于它占用物理内存，因此会减少可用于应用程序和数据处理的内存量。此外，RAM磁盘的内容在系统关机或重启时会丢失，因为内存是易失性的。
